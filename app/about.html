About this project

A learning project built to practise the full DevOps loop: write infrastructure as code, wire a pipeline, deploy automatically, then load test and observe it.

Request path

A visitor hits the Application Load Balancer, which spreads traffic across healthy EC2 instances in two Availability Zones. Each instance runs Nginx serving static files from /var/www/html. Only the load balancer's security group may reach port 80 on an instance — the instances are never addressed directly.

Infrastructure

VPC
A dedicated network with public subnets in two Availability Zones, an internet gateway and a shared route table.
Security groups
The ALB accepts HTTP from anywhere. Instances accept HTTP from the ALB's group only, and SSH from a single administrator address.
IAM
One instance role carrying just two managed policies — CloudWatch agent publishing and SSM access. No wildcards, no long-lived keys.
Launch template
Ubuntu 22.04 with IMDSv2 required, an encrypted gp3 root volume, and user data that installs Nginx and the CloudWatch agent.
Auto Scaling Group
Minimum 1, desired 2, maximum 4. Health is judged by the load balancer, so a failing web server is replaced even while the instance itself is fine.

Pipeline

  1. Validate — HTML is linted; Terraform is checked for formatting and validity.
  2. Plan — terraform plan runs and is posted back to the pull request.
  3. Build — the site is stamped with a commit SHA and timestamp, then uploaded as an artifact.
  4. Deploy — the artifact is copied to each instance over SSH and Nginx is reloaded.
  5. Verify — the load balancer is polled until it returns HTTP 200.

Security decisions

  • IMDSv2 is enforced, closing the SSRF path to instance credentials.
  • No credentials in the repository — the pipeline authenticates to AWS via OIDC.
  • Terraform state lives in a private, versioned, encrypted S3 bucket.
  • The backend bucket name is never hardcoded; it is supplied at init time.
